堵点（Blocker）是指在执行 MEU 之前**必须解决**的知识盲区或技术不确定性。正确识别堵点类型有助于制定高效的研究策略。

### 1.1 技术堵点

**定义**：不确定应该使用哪个 API、其参数、返回值或行为细节。

**识别信号**：
- 描述中包含"如何使用 XXX 的 YYY 功能？"
- 不确定某个库/框架的特定 API 是否存在或如何调用
- 对某个技术组件的内部工作机制不清楚

**典型示例**：
- "watchdog 的 `Observer` 类如何配置递归监控？`schedule()` 方法的 `recursive` 参数具体行为是什么？"
- "这个第三方支付 API 的请求格式是什么？签名算法是怎样的？"
- "SQLAlchemy 2.0 的 `select()` 和 `Query` 对象有什么区别？应该用哪个？"

**研究策略**：直接查阅官方文档的 API 参考部分，找到具体的类/方法定义和使用示例。

**搜索查询模板**：

| 查询类型 | 模板 | 示例 |
|---|---|---|
| API 用法 | `{库名} {功能名} 官方文档` | `watchdog Observer recursive 官方文档` |
| API 对比 | `{API_A} vs {API_B} {库名} 区别` | `select vs Query SQLAlchemy 2.0 区别` |

### 1.2 知识堵点

**定义**：存在多个可行方案，尚未做出技术选型决策。

**识别信号**：
- 描述中包含"A 方案还是 B 方案？"
- 需要对比多个技术方案的优劣
- 不确定业界最佳实践是什么

**典型示例**：
- "实时通知应该用 WebSocket、SSE 还是长轮询？"
- "ORM 应该用 SQLAlchemy 的 ORM 模式还是 Core 模式？"
- "前端状态管理应该用 Redux、Zustand 还是 Jotai？"

**研究策略**：搜索方案对比文章，重点关注：性能对比、社区活跃度、学习曲线、与项目技术栈的兼容性。

**搜索查询模板**：

| 查询类型 | 模板 | 示例 |
|---|---|---|
| 技术选型 | `{方案A} vs {方案B} {使用场景} 对比` | `WebSocket vs SSE 实时通知 对比` |
| 最佳实践 | `{技术} {场景} 最佳实践 {年份}` | `Python 文件监控 最佳实践 2024` |

### 1.3 工具堵点

**定义**：不确定应该使用哪个工具、库或框架，或者对所选工具的最佳实践不清楚。

**识别信号**：
- 描述中包含"用什么工具做 XXX？"
- 对某个工具的配置方式、插件生态不熟悉
- 不确定某个工具是否适合当前场景

**典型示例**：
- "Python 项目应该用 Poetry、pipenv 还是 uv 管理依赖？"
- "ESLint 和 Prettier 如何配置才能避免规则冲突？"
- "Docker Compose 的健康检查如何正确配置？"

**研究策略**：查阅工具的官方文档和 GitHub 仓库，关注最新版本的变化和推荐配置方式。

**搜索查询模板**：

| 查询类型 | 模板 | 示例 |
|---|---|---|
| 工具选型 | `{工具A} vs {工具B} {用途} 对比 {年份}` | `Poetry vs pipenv Python 依赖管理 对比 2024` |
| 配置指南 | `{工具名} {配置项} 配置教程` | `ESLint Prettier 配置冲突 解决方案` |

### 1.4 数据堵点

**定义**：缺乏执行任务所需的具体数据或规格信息。

**识别信号**：
- 描述中包含"数据格式是什么？""字段定义在哪里？"
- 需要了解外部系统的数据结构
- 缺少示例数据或测试数据

**典型示例**：
- "目标 API 的响应结构是什么？有哪些字段？"
- "数据库表的字段定义和约束是什么？"
- "CSV 文件的列分隔符是逗号还是制表符？编码是 UTF-8 还是 GBK？"

**研究策略**：查找 API 文档、数据库 Schema 定义、数据字典，或向相关方索要数据样本。

**搜索查询模板**：

| 查询类型 | 模板 | 示例 |
|---|---|---|
| API 数据结构 | `{API名} response schema 文档` | `Stripe API charge response schema` |
| 数据格式 | `{格式名} 规范 文档` | `CSV RFC 4180 规范` |

### 1.5 集成堵点

**定义**：不确定多个组件/系统之间的集成方式或兼容性。

**识别信号**：
- 描述中包含"A 系统和 B 系统如何对接？"
- 不确定接口协议、数据格式、认证方式
- 担心版本兼容性问题

**典型示例**：
- "这个 Node.js 库在 ARM 架构上能正常运行吗？"
- "PostgreSQL 在版本 14 中是否支持 XXX 特性？"
- "前端和后端的 API 接口如何保持同步？用什么工具管理 API 契约？"

**研究策略**：查阅兼容性矩阵、版本发布说明、集成指南。如有条件，创建最小化的集成原型进行验证。

**搜索查询模板**：

| 查询类型 | 模板 | 示例 |
|---|---|---|
| 兼容性 | `{库名} {目标环境} 兼容性` | `watchdog macOS Linux 兼容性` |
| 集成指南 | `{系统A} {系统B} 集成教程` | `Next.js Django API 集成教程` |

### 1.6 性能堵点

**定义**：不确定如何实现特定业务逻辑或算法以满足性能要求。

**识别信号**：
- 描述中包含"如何高效地实现 XXX？"
- 涉及大量数据处理、高并发、低延迟等场景
- 不确定某个算法的时间/空间复杂度是否满足要求

**典型示例**：
- "如何高效实现文件事件去重？在每秒数千个事件的情况下如何保证不丢事件？"
- "如何设计一个公平的任务调度算法？如何避免饥饿问题？"
- "如何在大文件上传时实现断点续传？分片大小如何确定？"

**研究策略**：搜索算法论文、性能优化博客、开源项目中的实现参考。关注时间/空间复杂度分析和基准测试数据。

**搜索查询模板**：

| 查询类型 | 模板 | 示例 |
|---|---|---|
| 性能优化 | `{技术} {场景} 性能优化` | `Python 高频文件事件 去重 性能` |
| 算法设计 | `{问题类型} 算法 设计` | `公平任务调度算法 避免饥饿` |
