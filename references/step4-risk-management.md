## 1. 风险识别来源

风险可以从以下来源识别：

| 来源 | 识别方法 |
|---|---|
| 研究结论 | 提取标记为"不确定"或"需要验证"的条目 |
| 评估备注 | 提取标记为"风险"或"注意事项"的条目 |
| 依赖分析 | 识别关键路径上的 MEU（关键路径上的 MEU 延误会直接影响项目进度） |
| 复杂度评估 | 评估得分低于 6.0 的 MEU 自动标记为高风险 |
| 开放问题 | `blocking` 级别的开放问题自动升级为风险 |

---

## 2. 风险严重度分级

| 严重度 | 定义 | 处理要求 |
|---|---|---|
| `critical` | 可能导致项目失败或需要重新设计 | **必须在计划生成后 24 小时内**制定缓解方案 |
| `high` | 可能导致 MEU 失败或大量返工 | 必须在对应 MEU 执行前制定缓解方案 |
| `medium` | 可能导致延迟或质量下降 | 建议在执行 Prompt 中提及，让执行者注意 |
| `low` | 影响较小，有成熟的缓解方案 | 记录在风险登记表中，按需处理 |

---

## 3. 开放问题的管理规则

### 3.1 关键规则

- **`critical` 级别的风险必须在计划生成后 24 小时内有缓解方案**
- **`blocking` 级别的开放问题必须在对应 MEU 执行前解决**
- **不得在存在未解决的 `critical` 风险时开始执行**

### 3.2 开放问题的处理流程

```
识别开放问题
    │
    ▼
评估影响级别
    │
    ├── blocking → 必须在执行前解决
    │               ├── 可以通过研究解决 → 添加研究任务
    │               ├── 需要外部决策 → 联系相关方
    │               └── 无法解决 → 调整任务范围或标记为风险
    │
    ├── risk → 在执行 Prompt 中标记，要求执行者注意
    │
    └── minor → 简要提及，允许执行者自行决定
```

## 5. 执行顺序 YAML 示例

```yaml
execution_order:
  - batch_number: 1
    meu_ids:
      - "1.1.1"
      - "1.2.1"
      - "1.2.2"
    can_parallel: true
    estimated_duration: "1.5-2h"
    notes: |
      1.1.1 是所有任务的基础，必须最先完成。
      1.2.1 和 1.2.2 只依赖 1.1.1，可以与 1.1.1 并行准备，
      但实际执行时需要等 1.1.1 完成后才能开始。

  - batch_number: 2
    meu_ids:
      - "1.1.2"
    can_parallel: true
    estimated_duration: "15-30min"
    notes: |
      1.1.2 依赖 1.1.1（需要项目结构已创建）。

  - batch_number: 3
    meu_ids:
      - "1.2.3"
    can_parallel: true
    estimated_duration: "1-2h"
    notes: |
      1.2.3 依赖 1.2.2（需要 FileEvent 数据类和事件队列）。

  - batch_number: 4
    meu_ids:
      - "1.2.4"
    can_parallel: true
    estimated_duration: "30min-1h"
    notes: |
      1.2.4 依赖 1.2.3（需要过滤后的事件流）。

  - batch_number: 5
    meu_ids:
      - "1.3.1"
    can_parallel: true
    estimated_duration: "1-2h"
    notes: |
      1.3.1 是集成任务，依赖 1.2.1（CLI）、1.2.3（过滤）、1.2.4（输出）。
      这是关键路径上的任务。

  - batch_number: 6
    meu_ids:
      - "1.4.1"
      - "1.4.2"
    can_parallel: true
    estimated_duration: "1-2h"
    notes: |
      1.4.1 和 1.4.2 都依赖 1.3.1（需要完整的集成代码）。
      两者可以并行执行。
```
