# 评估阶段 Prompt 模板

> **用途**: 基于研究结果对每个 MEU 进行可行性评估和方案选择
> **使用方法**: 将 `{research_results}` 替换为研究阶段的输出结果，直接复制使用
> **输入**: 研究阶段的完整输出（逐 MEU 研究报告 + 堵点汇总）
> **输出**: 每个 MEU 的评估分数、方案选择、设计决策和执行建议

---

## 模板

将以下内容复制后，替换 `{research_results}` 为研究阶段的实际输出：

````
# 任务：对 MEU 研究结果进行可行性评估

## 角色定义

你是一位资深技术评审专家和项目决策者。你的职责是基于研究阶段的技术调研结果，对每个 MEU 进行系统性的可行性评估，做出方案选择决策，并为后续的执行阶段提供明确的指导。

## 评估对象

以下是研究阶段产出的完整研究结果：

{research_results}

## 评估要求

### 评估维度

对每个 MEU，MUST 从以下 5 个维度进行评估，每个维度独立打分（0-10 分）：

#### 维度 1：技术可行性（Technical Feasibility）

评估该 MEU 在当前技术条件下是否可以成功实现。

| 分数 | 含义 |
|------|------|
| 9-10 | 技术方案成熟，有丰富的实现参考，几乎确定可以成功 |
| 7-8 | 技术方案可行，有少量不确定性，但风险可控 |
| 5-6 | 技术方案基本可行，但存在需要验证的假设 |
| 3-4 | 技术方案存在较大不确定性，需要进一步研究或替代方案 |
| 0-2 | 技术方案不可行，需要重新设计 |

#### 维度 2：复杂度可控性（Complexity Manageability）

评估该 MEU 的实现复杂度是否在可控范围内。

| 分数 | 含义 |
|------|------|
| 9-10 | 实现简单直接，逻辑清晰，几乎无复杂度 |
| 7-8 | 实现有一定复杂度，但结构清晰，易于管理 |
| 5-6 | 实现较复杂，需要仔细设计，但总体可控 |
| 3-4 | 实现复杂度高，需要大量设计工作，容易出错 |
| 0-2 | 实现极其复杂，超出单个 MEU 的处理能力 |

#### 维度 3：依赖风险（Dependency Risk）

评估该 MEU 的依赖项是否可靠、可用。

| 分数 | 含义 |
|------|------|
| 9-10 | 无外部依赖或依赖完全可靠 |
| 7-8 | 依赖基本可靠，有成熟的替代方案 |
| 5-6 | 依赖存在一定风险，但有缓解措施 |
| 3-4 | 依赖风险较高，可能影响执行 |
| 0-2 | 依赖不可靠，严重阻塞执行 |

#### 维度 4：堵点影响（Blocker Impact）

评估已识别的堵点对 MEU 执行的影响程度。

| 分数 | 含义 |
|------|------|
| 9-10 | 无堵点或堵点影响极小 |
| 7-8 | 有轻微堵点，有明确的解决方案 |
| 5-6 | 有中等堵点，需要额外工作来克服 |
| 3-4 | 有严重堵点，可能导致大幅延期或返工 |
| 0-2 | 有致命堵点，可能导致 MEU 无法完成 |

#### 维度 5：执行清晰度（Execution Clarity）

评估该 MEU 的执行路径是否清晰明确。

| 分数 | 含义 |
|------|------|
| 9-10 | 执行路径非常清晰，可直接按步骤实现 |
| 7-8 | 执行路径基本清晰，少量细节需要在执行中确定 |
| 5-6 | 执行路径大致明确，需要在执行中做若干设计决策 |
| 3-4 | 执行路径不够清晰，需要进一步研究或设计 |
| 0-2 | 执行路径完全不明确，需要重新规划 |

### 综合评分

综合评分 MUST 按以下加权公式计算：

```
综合评分 = 技术可行性 × 0.30 + 复杂度可控性 × 0.25 + 依赖风险 × 0.20 + 堵点影响 × 0.15 + 执行清晰度 × 0.10
```

综合评分 MUST 四舍五入到一位小数。

### 评估等级

根据综合评分，MUST 为每个 MEU 分配评估等级：

| 综合评分 | 评估等级 | 含义 | 后续动作 |
|----------|----------|------|----------|
| 8.5-10.0 | **A - 就绪** | 可直接进入执行阶段 | 无需额外准备 |
| 7.0-8.4 | **B - 基本就绪** | 可执行，有少量注意事项 | 记录注意事项 |
| 5.0-6.9 | **C - 有条件就绪** | 可执行，需要先解决特定问题 | MUST 先解决标注的问题 |
| 3.0-4.9 | **D - 需要调整** | 当前方案不可行，需要调整 | MUST 修改方案后重新评估 |
| 0.0-2.9 | **F - 不可行** | 无法在当前条件下执行 | MUST 重新设计或放弃 |

### 方案选择

对于研究阶段提供了多个技术方案的 MEU，MUST 明确选择一个推荐方案：

- MUST 说明选择理由
- MUST 说明未选方案的排除原因
- SHOULD 标注选择的不确定性程度

### Benchmark 设计原则

对于复杂度评估为 medium 或 high 的 MEU，MUST 设计验收 Benchmark：

1. **可量化**：Benchmark MUST 包含具体的数值指标（如响应时间、吞吐量、内存占用）
2. **可复现**：Benchmark MUST 提供可复现的测试条件（输入数据、环境配置）
3. **有基线**：Benchmark MUST 提供参考基线值（来自研究阶段的性能数据或行业标准）
4. **分场景**：Benchmark SHOULD 覆盖正常场景和边界场景

Benchmark 输出格式：
```markdown
#### 验收 Benchmark

| 指标 | 正常场景 | 边界场景 | 基线参考 |
|------|----------|----------|----------|
| [指标名] | [目标值] | [目标值] | [来源] |

**测试条件**:
- 输入数据: [描述]
- 环境配置: [描述]
- 测试方法: [描述]
```

```markdown

- **B 级（基本就绪）**: X 个

```

对每个 MEU，输出以下结构的评估报告：

| 维度 | 分数 | 权重 | 加权分 | 评分理由 |

|--------|------|----------|------|----------|----------|

```

## 质量检查

在输出最终结果之前，MUST 逐项确认以下检查清单：

- [ ] 每个 MEU 都有完整的 5 维度评分
- [ ] 综合评分计算正确（加权公式）
- [ ] 评估等级分配正确（基于综合评分区间）
- [ ] D/F 级别的 MEU 提供了调整建议
- [ ] C 级别的 MEU 标注了前置条件
- [ ] 有多个方案的 MEU 做出了明确选择
- [ ] medium/high 复杂度的 MEU 设计了验收 Benchmark
- [ ] Benchmark 指标可量化、可复现、有基线
- [ ] 评估汇总表信息完整

## 约束条件

- MUST 使用中文输出
- MUST 保持评估的客观性和一致性（相同条件下的 MEU 应获得相近评分）
- MUST 基于研究结果进行评估，NEVER 引入研究结果之外的信息作为评估依据
- SHOULD 对评分较低的项目提供建设性的改进建议
- MUST NOT 因为单个维度的低分而忽略其他维度的优势（使用加权综合评分）
- MUST NOT 对所有 MEU 都给出高分（评估 MUST 有区分度）
````

## 使用说明

### 如何使用此模板

1. **前置条件**：MUST 先完成研究阶段，获得完整的 MEU 研究报告
2. **复制**：复制上方代码块中的全部内容
3. **替换占位符**：将 `{research_results}` 替换为研究阶段的完整输出
4. **粘贴执行**：将替换后的内容粘贴到 LLM 对话窗口中执行
5. **检查输出**：对照质量检查清单验证输出质量

### 填写 {research_results} 的建议

`{research_results}` SHOULD 包含研究阶段的完整输出，特别是：
- 研究总览
- 逐 MEU 研究报告（推荐方案、堵点分析、研究结论）
- 堵点汇总表
- 依赖关系补充（如有）

### 评估结果的使用

评估阶段的输出将直接用于：
1. **执行计划生成**：评估等级决定 MEU 是否可以进入执行阶段
2. **执行 Prompt 生成**：方案选择和设计决策将写入执行 Prompt 的上下文信息
3. **风险管理**：关键风险将记录在执行计划的风险登记中
4. **资源分配**：综合评分和复杂度评估影响执行优先级

### 与其他阶段的关系

```
研究阶段                    评估阶段                    计划生成
┌──────────┐              ┌──────────┐              ┌──────────┐
│ 技术方案   │──方案对比──→│ 可行性评估 │──评估结论──→│ 执行计划   │
│ 堵点分析   │──堵点汇总──→│ 方案选择   │──设计决策──→│ 执行 Prompt │
│ 性能数据   │──基准数据──→│ Benchmark │──验收标准──→│ 验收标准   │
└──────────┘              └──────────┘              └──────────┘
```
