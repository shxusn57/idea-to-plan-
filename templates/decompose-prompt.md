# 拆解阶段 Prompt 模板

> **用途**: 将原始想法/需求拆解为可执行的最小执行单元（MEU）树
> **使用方法**: 将 `{idea}` 替换为实际的想法描述，直接复制使用
> **输入**: 一个产品想法、功能需求或项目描述
> **输出**: 结构化的 MEU 任务树

---

## 模板

将以下内容复制后，替换 `{idea}` 为你的实际想法：

````
# 任务：将想法拆解为最小执行单元（MEU）树

## 角色定义

你是一位资深技术架构师和项目管理专家。你擅长将模糊的想法转化为清晰、可执行的任务分解结构。你的拆解结果将直接用于指导后续的研究、评估和执行阶段。

## 原始想法

{idea}

## 拆解要求

### 核心原则

1. **MEU 定义**：每个最小执行单元（MEU）MUST 满足以下全部条件：
   - 可独立执行：一个 MEU 可以由一个人或一个 LLM 会话独立完成
   - 可独立验证：MEU 完成后有明确的验收标准
   - 粒度适中：一个 MEU 的预估执行时间在 30 分钟到 4 小时之间
   - 边界清晰：MEU 之间无重叠，职责明确

2. **拆解深度**：
   - MUST 拆解到叶子节点全部为 MEU 级别
   - 中间节点为功能模块或逻辑分组
   - 树的深度建议控制在 3-5 层
   - NEVER 拆解到过于琐碎的步骤（如"创建文件"、"写一行代码"）

3. **完整性**：
   - MUST 覆盖从项目启动到交付的全部工作
   - MUST 包含测试、文档、配置等非功能性工作
   - SHOULD 包含环境搭建和依赖安装（如适用）
   - SHOULD 包含最终集成验证

### 拆解维度

MUST 从以下维度进行拆解：

| 维度 | 说明 | 示例 |
|------|------|------|
| **功能模块** | 按功能职责划分 | 用户认证、数据处理、UI 渲染 |
| **技术层次** | 按技术栈层次划分 | 数据库层、服务层、接口层、前端层 |
| **生命周期** | 按开发阶段划分 | 设计、实现、测试、部署 |
| **质量保障** | 按质量活动划分 | 单元测试、集成测试、性能测试、文档 |

### 依赖关系

- MUST 为每个 MEU 标注其依赖的其他 MEU
- 依赖关系 MUST 基于技术依赖（产出物被消费）或逻辑依赖（设计决策被依赖）
- SHOULD 识别可并行执行的 MEU
- NEVER 创建循环依赖

### 命名规范

- MEU ID 格式：`MEU-XXX`（三位数字，从 001 开始）
- MEU 名称格式：动词开头 + 名词短语（如"实现用户认证模块"、"设计数据库 Schema"）
- 中间节点名称：名词短语（如"用户认证"、"数据持久化"）

## 输出格式

### 1. 任务树结构（MUST 输出）

使用缩进列表表示树形结构，每个节点包含 ID、名称和简要描述：

```
项目名称
├── 模块 A
│   ├── MEU-001: 任务名称
│   │   - 描述：...
│   │   - 依赖：无
│   │   - 预估复杂度：low/medium/high
│   ├── MEU-002: 任务名称
│   │   - 描述：...
│   │   - 依赖：MEU-001
│   │   - 预估复杂度：medium
│   └── 模块 A.1
│       └── MEU-003: 任务名称
│           - 描述：...
│           - 依赖：MEU-001, MEU-002
│           - 预估复杂度：high
├── 模块 B
│   └── ...
└── 质量保障
    └── ...
```

### 2. MEU 明细表（MUST 输出）

以表格形式列出所有叶子节点 MEU：

| MEU ID | 名称 | 描述 | 依赖 | 所属模块 | 预估复杂度 | 预估工时 |
|--------|------|------|------|----------|------------|----------|
| MEU-001 | ... | ... | 无 | ... | medium | 2h |
| MEU-002 | ... | ... | MEU-001 | ... | high | 3h |

### 3. 依赖关系摘要（MUST 输出）

列出所有依赖关系，用于后续拓扑排序：

```yaml
dependencies:
  MEU-001: []
  MEU-002: [MEU-001]
  MEU-003: [MEU-001, MEU-002]
  # ...
```

### 4. 拆解说明（SHOULD 输出）

简要说明拆解思路、关键决策和注意事项：

- **拆解策略**：采用了什么拆解策略（按功能模块 / 按技术层次 / 混合策略）
- **关键决策**：哪些地方做了拆分或合并的决策，以及原因
- **风险提示**：哪些 MEU 存在不确定性或潜在风险
- **建议执行顺序**：基于依赖关系的建议执行批次

## 质量检查

在输出最终结果之前，MUST 逐项确认以下检查清单：

- [ ] 每个 MEU 可独立执行（一个人或一个 LLM 会话可完成）
- [ ] 每个 MEU 有明确的验收标准（可在输出中简要描述）
- [ ] 每个 MEU 的预估工时在 30 分钟到 4 小时之间
- [ ] MEU 之间无职责重叠
- [ ] 依赖关系无循环
- [ ] 覆盖了从设计到测试到交付的完整生命周期
- [ ] 包含了非功能性工作（测试、文档等）
- [ ] 树的深度在 3-5 层之间
- [ ] 命名符合规范

## 约束条件

- MUST 使用中文输出
- MUST 保持客观、专业的语气
- MUST NOT 添加原始想法中未提及的功能需求（除非是隐含的必要工作）
- SHOULD 在不确定时标注"需要进一步研究"
- SHOULD 为每个 MEU 提供足够详细的描述，使后续阶段无需回溯原始想法
````

## 使用说明

### 如何使用此模板

1. **复制**：复制上方代码块中的全部内容
2. **替换占位符**：将 `{idea}` 替换为你的实际想法描述
3. **粘贴执行**：将替换后的内容粘贴到 LLM 对话窗口中执行
4. **检查输出**：对照质量检查清单验证输出质量

### 填写 {idea} 的建议

`{idea}` SHOULD 包含以下信息以获得最佳拆解结果：

| 信息项 | 是否必须 | 说明 |
|--------|----------|------|
| 核心功能描述 | MUST | 想法要解决什么问题、实现什么功能 |
| 目标用户/场景 | SHOULD | 谁会使用、在什么场景下使用 |
| 技术偏好 | SHOULD | 偏好的编程语言、框架、工具 |
| 约束条件 | SHOULD | 时间、资源、技术限制 |
| 参考项目 | 可选 | 类似的现有项目或产品 |

### {idea} 填写示例

```
{idea} = 我想做一个文件监控 CLI 工具，使用 Python 编写。
功能包括：监控指定目录的文件变化（创建、修改、删除），
支持按文件类型和路径模式过滤事件，支持去重，
输出格式支持 JSON 和人类可读的表格格式。
要求使用 Python 标准库或常用第三方库，
不需要 GUI，纯命令行工具。
```
