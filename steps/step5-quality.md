## 1. 两套评估体系概览

本框架包含两套互补的评估体系：

| 评估体系 | 目的 | Benchmark 数量 | 评估方法 | 通过标准 |
|---------|------|---------------|---------|---------|
| Prompt 执行合规性评估 | 检查最终输出是否准确执行了各阶段 Prompt | 23 | LLM-as-judge Yes/No | 合规率 >= 90% |
| Plan 质量评估 | 检查计划的可执行性、研究充分性、验收标准质量、计划完整性 | 15 | LLM-as-judge Rubric 评分 | 加权平均 >= 4.0/5.0 |

## 2. 合规性评估 Benchmark 汇总表

共 23 个 Benchmark，按 4 个阶段分组，LLM-as-judge 逐项 Yes/No 判定。

### Step 1 拆解合规性（4 个）

| 编号 | Benchmark | 检查内容 | 目标 |
|------|-----------|---------|------|
| B1.1 | 节点格式合规率 | 所有节点包含全部必填字段（id、name、objective/description、parent、children、dependencies、is_meu/type） | 100% |
| B1.2 | MEU 判定合规率 | 所有叶子节点满足 4 个 MEU 条件（目标单一、可独立完成、产出可验证、依赖可满足） | 100% |
| B1.3 | 依赖关系正确率 | 跨分支依赖引用正确，父子依赖未重复 | 100% |
| B1.4 | 命名规范合规率 | 所有叶子节点名称符合"动词+名词"格式（仅检查叶子节点） | >= 90% |

| 编号 | Benchmark | 检查内容 | 目标 |
|------|-----------|---------|------|
| B2.1 | 研究覆盖率 | 所有叶子节点都有对应的研究结果 | 100% |
| B2.2 | 堵点方案数量 | 每个堵点 >= 2 个候选方案，且有 pros/cons 分析 | 100% |
| B2.3 | 方案选择完整性 | 每个堵点有明确的选择结果和选择理由（reasoning） | 100% |
| B2.4 | 无堵点节点处理 | 无堵点节点记录了 key_findings 且 >= 1 | 100% |
| B2.5 | 置信度标注率 | 所有研究结论标注了置信度（high/medium/low） | 100% |
| B2.6 | 研究轮次完整性 | 每个 MEU 完成 4 轮研究流程（初始分析、定向搜索、交叉验证、结论整合） | 100% |
| B2.7 | 信息源多样性 | 每个堵点 >= 2 个独立信息源 | 100% |
| B2.8 | 决策摘要完整性 | 每个 MEU 有 decision_summary（含 key_decisions 和 pitfalls_overview） | 100% |
| B2.9 | 踩坑记录 | 有堵点的 MEU 记录了踩坑信息（pitfalls） | >= 80% |

## 4. 通过标准

### 合规性评估

- **通过**：23 个 Benchmark 中合规率 >= 90%（即至少 21/23 通过）
- **不通过**：合规率 < 90%

### 质量评估

- **通过**：4 个维度的加权平均分 >= 4.0/5.0
  - 可执行性 (30%) + 研究充分性 (25%) + 验收标准质量 (25%) + 计划完整性 (20%)
- **不通过**：加权平均分 < 4.0/5.0

### 总体判定

- **最终通过**：合规性评估 AND 质量评估均通过
- **最终不通过**：任一评估不通过

## 5. Checklist

评估前，确认以下准备工作：

- [ ] 任务树文件已生成（task-tree.yaml）
- [ ] 研究结果文件已生成（research.yaml）
- [ ] 评估标准文件已生成（evaluation.yaml）
- [ ] 执行计划文件已生成（execution-plan.md）
- [ ] 评估配置已就绪（evaluation-config.yaml）
- [ ] LLM-as-judge 使用与计划生成不同的模型实例
- [ ] 评估结果将保存为 YAML 格式
