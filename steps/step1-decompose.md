## 1. MEU 的四个必要条件

MEU（Minimum Executable Unit，最小可执行单元）是任务树的**叶子节点**，代表最终交付给执行者完成的最小工作单元。一个节点要被标记为叶子节点（即 MEU），**必须同时满足以下四个条件**。任何一个条件不满足，都需要继续拆解。

### 1.1 单一职责

**定义**：一个 MEU 只负责完成一件明确的事情，产生一个核心交付物。

**判断标准**：
- 描述中是否包含"和"、"并且"、"同时"、"以及"等连接词？如果有，很可能是多个职责的混合。
- 该任务是否可以用一句话概括其核心目标？如果不能，说明职责过多。
- 完成该任务后，是否只产生一个主要产出物？如果产出多个不相关的文件/模块，说明需要拆分。

**实操建议**：对每个叶子节点的描述做"连接词检测"——搜索"和"、"并且"、"同时"、"以及"、"also"、"and"、"meanwhile"等关键词，每发现一个就评估是否需要拆分。

### 1.2 边界清晰

**定义**：任务的输入、输出和范围被完整定义，任何两个合格的执行者看到该描述后，应该产出高度相似的结果。

**判断标准**：
- 输入是否明确？（从哪里获取数据/参数/依赖？）
- 输出是否明确？（产出什么文件/接口/配置？具体格式是什么？）
- 范围是否明确？（做什么、不做什么是否界定清楚？）
- 是否存在歧义？（描述是否可能被不同人做不同理解？）

**实操建议**：对每个叶子节点的描述做"疑问句检测"——如果执行者看完描述后还需要问"具体用什么技术？""输出什么格式？""包含哪些功能？"等问题，说明边界不够清晰。

### 1.3 可独立执行

**定义**：执行者仅凭 MEU 自身的描述以及其依赖任务的研究结果，就能完成该任务，不需要额外的上下文或口头沟通。

**判断标准**：
- 是否引用了外部文档但没有提供关键信息摘要？
- 是否依赖尚未完成的其他任务的产出？
- 是否需要执行者自行做技术选型或架构决策？
- 描述中是否包含"参考 XXX 文档"、"按照 XXX 的设计"等指向外部信息的表述？

**实操建议**：采用"新人测试"——假设一个完全不了解项目背景的新人拿到这个 MEU 描述，他能否仅凭描述和依赖研究结果完成任务？如果不能，需要补充信息或继续拆解。

### 1.4 粒度适当

**定义**：任务可以在一次 AI 会话中完成（大约 30 分钟到 4 小时的工作量），既不过大（需要重新拆分），也不过小（管理成本超过执行价值）。

**判断标准**：
- **过大**：需要跨越多个技术领域、涉及大量文件修改、需要多轮迭代才能完成、无法在一次会话中验证。
- **过小**：只是修改一行配置、添加一个简单的常量定义、管理该任务的成本（描述、评估、跟踪）比执行本身还高。

**粒度参考指南**：

| 复杂度 | 预估工作量 | 典型特征 |
|---|---|---|
| trivial | < 30 分钟 | 简单配置文件创建、常量定义 |
| low | 30 分钟 - 1 小时 | 单一功能实现、简单函数编写 |
| medium | 1 - 2 小时 | 需要设计思考、涉及多个组件协调 |
| high | 2 - 4 小时 | 复杂业务逻辑、需要架构设计、多系统交互 |
| extreme | > 4 小时 | 高技术风险、需要原型验证、涉及性能/安全 |

**实操建议**：如果预估工作量超过 4 小时，几乎可以确定需要继续拆解。如果预估工作量少于 15 分钟，考虑是否可以与相邻任务合并。

## 2. 四个拆解维度

对每个非叶子节点，从以下四个维度进行分析，判断是否需要继续拆解。**一个节点只要在任何一个维度上需要拆解，就应该继续拆解**。

### 2.1 功能维度拆解

**核心问题**：这个任务包含多少个**独立的功能点**？每个功能点是否可以独立交付和测试？

**分析方法**：列出任务描述中提到的所有功能点 → 判断每个功能点是否可以独立实现和验证 → 如果有多个独立功能点，按功能拆分。

**判断要点**：
- 每个子功能是否有明确的输入和输出？
- 移除其中一个子功能，其他子功能是否仍然有意义？
- 每个子功能是否可以独立测试？

### 2.2 流程维度拆解

**核心问题**：这个任务包含多少个**有先后顺序的阶段**？每个阶段是否有明确的阶段产出？

**分析方法**：梳理任务的执行流程 → 识别流程中的关键阶段节点 → 每个阶段是否有明确的输入（上一阶段的产出）和输出（下一阶段的输入）。

**判断要点**：
- 各阶段之间是否有明确的先后依赖关系？
- 每个阶段完成后是否有一个可验证的中间产出？
- 是否存在可以并行的阶段？（如果有，说明它们是独立的，应该拆分）

### 2.3 技术维度拆解

**核心问题**：这个任务涉及多少个**不同的技术领域**？每个领域是否需要不同的专业知识？

**分析方法**：列出任务涉及的所有技术栈和领域 → 判断每个技术领域是否需要独立的设计和实现 → 如果涉及多个差异较大的技术领域，按技术拆分。

**判断要点**：
- 各技术领域是否需要不同的专业知识？
- 各技术领域的实现是否可以独立进行（通过接口解耦）？
- 某个技术领域的变更是否会影响其他领域？（如果是，需要通过接口约束来解耦）

### 2.4 知识维度拆解

**核心问题**：这个任务存在哪些**知识盲区**？这些盲区是否需要独立的研究和验证？

**分析方法**：识别任务中不确定的技术选型、API 用法、最佳实践 → 判断这些知识盲区是否可以独立研究 → 如果存在多个独立的知识盲区，为每个盲区创建独立的研究任务。

**判断要点**：
- 哪些技术决策需要先研究才能确定？
- 研究结果是否会影响后续的实现方式？
- 是否存在多个互不依赖的研究方向？（如果是，可以并行研究）

## 3. 拆解终止条件

当以下**任意一个条件**成立时，应该停止对该节点的拆解：

| 条件 | 详细说明 | 判断方法 |
|---|---|---|
| **MEU 四条件全部满足** | 节点同时满足单一职责、边界清晰、可独立执行、粒度适当 | 逐一检查四个条件 |
| **无法继续拆分** | 进一步拆分只会产生无意义的碎片（如"定义一个变量"、"写一行注释"） | 拆分后的子任务是否仍有独立价值？ |
| **边际收益递减** | 子任务的管理成本（描述、评估、跟踪）已经超过拆分带来的执行便利 | 管理成本 > 执行便利性 |
| **信息不足** | 需要先进行研究才能决定如何拆分 | 标记为"需要研究"，暂停拆解 |

**决策流程**：

```
该节点是否满足 MEU 四个条件？
  是 → 标记为叶子节点，停止拆解
  否 → 是否有足够的信息来决定如何拆分？
    是 → 继续拆解
    否 → 标记为"需要研究"，暂停拆解，先进行研究
```

**重要约束**：
- 任务树的深度应控制在 **3-5 层**以内。如果超过 5 层，说明顶层设计可能有问题，需要重新审视根节点的划分方式。
- 同一层级的节点粒度应该大致相当。如果同一层级中有的节点很大、有的很小，说明拆解不均匀。

## 4. 节点格式

任务树中的每个节点使用 YAML 格式描述，核心字段如下：

```yaml
- id: "1.2.3"
  name: "实现 CLI 参数解析模块"
  description: |
    使用 Python 标准库 argparse 实现 CLI 参数解析：
    - --path（必填）：监控的目录路径
    - --recursive（可选，默认 True）：是否递归监控子目录
    输出文件：src/filewatcher/cli.py
  type: leaf
  parent: "1.2"
  children: []
  dependencies: ["1.1.1"]
  priority: critical
  estimated_complexity: low
  status: pending
```

**字段速查**：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | 是 | 节点唯一标识符，格式为 `{父节点id}.{序号}` |
| `name` | string | 是 | 简洁的任务名称，不超过 50 字符，建议"动词 + 名词" |
| `description` | string | 是 | 详细任务描述，包含输入、输出、技术要求、验收标准 |
| `type` | string | 是 | `root` / `branch` / `leaf` |
| `parent` | string\|null | 是 | 父节点 ID，根节点为 `null` |
| `children` | string[] | 是 | 子节点 ID 列表，叶子节点为空数组 |
| `dependencies` | string[] | 是 | 跨分支依赖的节点 ID 列表（不含父子依赖） |
| `priority` | string | 是 | `critical` / `high` / `medium` / `low` |
| `estimated_complexity` | string | 是 | `trivial` / `low` / `medium` / `high` / `extreme` |
| `status` | string | 是 | `pending` / `researching` / `ready` / `in_progress` / `completed` / `blocked` |

> 完整字段参考（含优先级/复杂度/状态详细指南）请参见 [step1-node-reference.md](../references/step1-node-reference.md)。

## 5. 核心规则

1. **MEU 四条件缺一不可**：单一职责、边界清晰、可独立执行、粒度适当必须全部满足。
2. **任何维度触发即拆分**：功能、流程、技术、知识四个维度中，任何一个维度需要拆分就继续拆分。
3. **依赖只声明跨分支**：父子依赖是隐式的，`dependencies` 字段只填写跨分支依赖。
4. **深度控制在 3-5 层**：超过 5 层需重新审视根节点划分。
5. **同层粒度大致相当**：同一层级节点的预估工作量差异不应超过 2 倍。
6. **拆解阶段回答"做什么"**：研究阶段回答"怎么做"。如果拆解时发现需要技术细节，标记为"需要研究"。

## 6. Checklist

拆解完成后，逐项检查：

- [ ] 所有叶子节点同时满足 MEU 四个条件（单一职责、边界清晰、可独立执行、粒度适当）
- [ ] 所有叶子节点描述中不包含"和"、"并且"、"同时"等连接词（或已确认不需要拆分）
- [ ] 所有叶子节点的输入、输出、范围被明确定义
- [ ] 所有叶子节点可由执行者仅凭描述和依赖研究结果独立完成
- [ ] 所有叶子节点的预估工作量在 30 分钟到 4 小时之间
- [ ] 任务树深度在 3-5 层以内
- [ ] 同一层级节点粒度大致相当
- [ ] 所有节点包含全部必填字段（id、name、description、type、parent、children、dependencies、priority、estimated_complexity、status）
- [ ] 跨分支依赖引用的节点 ID 存在且确实属于不同分支
- [ ] 父子依赖未在 `dependencies` 字段中重复声明
- [ ] 依赖图中不存在循环依赖
- [ ] 没有遗漏隐含任务（项目初始化、配置、测试、文档、错误处理等）
- [ ] 拆解描述使用业务语言而非技术实现语言
