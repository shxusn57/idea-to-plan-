## 1. Benchmark 设计原则

Benchmark（验收基准）是评估 MEU 执行结果是否合格的**客观标准**。每个 MEU 必须配备一组 Benchmark，所有 Benchmark 全部通过才能认为该 MEU 执行成功。

### 1.1 全部达标原则

**定义**：一个 MEU 的所有 Benchmark 必须**全部通过**，才能认为该 MEU 执行成功。不存在"可选"或"加分"的 Benchmark。

**实操建议**：如果某个检查项确实不是必须的，直接不要把它作为 Benchmark。Benchmark 列表中的每一项都必须是硬性要求。

### 1.2 可量化原则

**定义**：每个 Benchmark 的通过条件必须是**客观、可量化**的，不能依赖主观判断。优先使用以下类型的判定标准：
- 布尔判定（是/否）：文件是否存在、函数是否定义
- 数值比较（>=, <=, ==）：覆盖率 >= 80%、响应时间 <= 200ms
- 存在性检查：代码中是否包含特定的函数调用、配置项

**实操建议**：对每个 Benchmark 描述做"量化检测"——问自己"一个不了解背景的人能否仅凭这个描述客观判断通过/不通过？"如果不能，需要改写。

### 1.3 可测试原则

**定义**：Benchmark 的验证方式应该**优先使用命令行可验证**的方法。如果需要手动验证，必须提供明确的操作步骤。

**优先级排序**（从高到低）：
1. **自动化测试**：可以通过脚本或测试框架自动验证（最佳）
2. **命令行检查**：可以通过单条命令验证（如 `grep`、`test -f`）
3. **代码审查**：需要人工阅读代码进行验证
4. **手动操作**：需要人工执行特定操作并观察结果（最后手段）

### 1.4 与研究关联原则

**定义**：如果研究阶段推荐了特定的技术方案，Benchmark 必须**验证该方案是否被采纳**。如果研究阶段识别了特定的注意事项或陷阱，Benchmark 必须**覆盖这些注意事项**。

**具体要求**：
- 研究推荐了方案 A → Benchmark 中必须有"使用了方案 A"的验证
- 研究指出了 API X 的注意事项 → Benchmark 中必须有覆盖该注意事项的测试用例
- 研究的置信度为 `low` → Benchmark 中应该包含验证步骤

### 1.5 无遗漏原则

**定义**：Benchmark 集合必须**覆盖 MEU 的所有交付物**。除了明显的代码文件外，还要检查隐含的交付物。

**常见被遗漏的交付物**：
- 配置文件（`.env.example`、`config.yaml`）
- 类型定义文件（`types.ts`、`models.py`）
- 数据库迁移文件
- 导出/入口文件（`__init__.py`、`index.ts`）
- 文档更新（README、CHANGELOG）
- 测试文件

**实操建议**：对照 MEU 的 `description` 字段，逐条列出所有提到的交付物，然后为每个交付物创建至少一个 Benchmark。

## 2. Benchmark 类型概览

| 类型 | 名称 | 说明 |
|---|---|---|
| `file_existence` | 文件存在性 | 验证预期的文件或目录是否被创建 |
| `content_correctness` | 内容正确性 | 验证文件内容是否包含预期的代码、配置或数据 |
| `functional_test` | 功能测试 | 通过运行代码或命令来验证运行时行为 |
| `quality_standard` | 质量标准 | 验证代码质量、测试覆盖率、性能等非功能性指标 |
| `compatibility_verification` | 兼容性验证 | 验证多组件协作和端到端工作流 |

> 每种类型的完整详解、YAML 格式和设计要点请参见 [step3-benchmark-types.md](../references/step3-benchmark-types.md)。

## 3. 禁止模式

以下描述模式**禁止使用**。如果发现类似描述，必须改写为可量化的版本。

| 禁止的描述 | 问题 | 改写为 |
|---|---|---|
| "代码质量很好" | 完全主观，无法量化 | "`ruff check` 零错误，`ruff format --check` 零差异" |
| "功能正常工作" | "正常"没有定义 | "输入 X 产生 Y，输入 Z 返回错误 E" |
| "性能可以接受" | 没有性能指标 | "处理 1000 条记录耗时 < 2 秒" |
| "UI 看起来不错" | 纯主观判断 | "在 320px/768px/1024px 宽度下无溢出" |
| "基本完成" | "基本"意味着可能没完成 | 列出具体的完成标准清单 |
| "遵循最佳实践" | 没有说明是哪个实践 | "遵循 {具体规范} 的第 {X} 条规则" |
| "错误处理完善" | 没有说明处理了哪些错误 | "场景 A 返回错误码 E1，场景 B 记录日志并继续" |
| "代码结构清晰" | 主观判断 | "每个函数不超过 30 行，模块间无循环依赖" |
| "文档完整" | 没有说明包含哪些内容 | "README 包含安装、使用、开发三个章节" |
| "测试充分" | 没有说明覆盖范围 | "单元测试覆盖率 >= 80%，覆盖所有公共 API" |
| "兼容性良好" | 没有说明兼容什么 | "在 Python 3.11 和 3.12 上测试通过" |
| "安全性足够" | 没有安全标准 | "通过 bandit 扫描零高危漏洞，无 SQL 注入风险" |

**自检方法**：对每个 Benchmark 描述问以下问题：
1. 两个不同的人独立评估，是否会给出相同的结果？
2. 是否存在任何歧义或多种理解方式？
3. 是否需要额外的上下文才能理解"通过"的标准？

如果任何一个问题的答案是"否"或"需要"，该描述就需要改写。

## 4. Evaluation 概要

Evaluation（评估）是对 MEU 执行结果进行**系统性验证**的过程。每个 MEU 执行完成后，必须进行评估。

### 评估方法

根据 Benchmark 类型选择合适的评估方法：

| Benchmark 类型 | 推荐评估方法 | 推荐工具 |
|---|---|---|
| `file_existence` | 自动化脚本 | `test -f`、`os.path.exists()`、`ls` |
| `content_correctness` | 自动化 + 人工审查 | `grep`、AST 解析、代码审查 |
| `functional_test` | 自动化测试 | `pytest`、`jest`、单元测试框架 |
| `quality_standard` | 自动化工具 | linter（ruff/eslint）、coverage 工具 |
| `compatibility_verification` | 自动化 + 手动测试 | E2E 测试框架、手动操作验证 |

### 评估步骤

对每个 MEU 的评估按以下步骤执行：

1. **准备阶段**：确认所有依赖的 Benchmark 工具已安装；确认测试环境已准备就绪；阅读 MEU 的 Benchmark 列表
2. **执行阶段**：按 Benchmark ID 顺序逐个执行验证；记录每个 Benchmark 的通过/不通过状态；如果不通过，记录具体的不通过原因
3. **汇总阶段**：统计通过和不通过的 Benchmark 数量；如果存在不通过的 Benchmark，进行根因分析；编写评估报告

### 通过条件

一个 MEU 的评估结果为"通过"，**必须同时满足以下所有条件**：

1. **所有 Benchmark 全部通过**：每个 Benchmark 的检查结果都是 `pass`
2. **无回归**：没有引入新的编译错误、运行时错误或功能退化
3. **交付物完整**：所有预期的文件、代码、文档都已产出
4. **与研究一致**：实现方案与研究推荐一致（如有偏差，必须有文档化的理由）

### 失败处理

当 MEU 评估不通过时：

1. **根因分析**：确定失败的根本原因（实现错误 / Benchmark 设计缺陷 / 研究结论有误 / 需求变更 / 依赖 MEU 有问题）
2. **记录和跟踪**：在评估日志中记录失败详情；将 MEU 状态更新为 `failed`；创建修复任务
3. **修复和重新评估**：根据根因分析结果执行修复；修复完成后重新运行评估；重复此过程直到所有 Benchmark 通过

> 详细的失败处理流程和评估报告格式请参见 [step3-benchmark-types.md](../references/step3-benchmark-types.md)。

## 5. Checklist

评估标准设计完成后，逐项检查：

- [ ] 每个 MEU 至少有 2 个 Benchmark
- [ ] 所有 Benchmark 均为必通过项，无"可选"或"加分"项
- [ ] 每个 Benchmark 的通过条件客观、可量化
- [ ] 每个 Benchmark 标注了验证方式（自动化测试 / 命令行检查 / 代码审查 / 手动操作）
- [ ] Benchmark 覆盖了研究阶段推荐的技术方案和注意事项
- [ ] Benchmark 覆盖了 MEU 的所有交付物（包括隐含交付物）
- [ ] Benchmark 描述中不包含禁止的模糊描述
- [ ] 每个 MEU 有明确的通过条件（所有 Benchmark 通过 + 无回归 + 交付物完整 + 与研究一致）
- [ ] 每个 Benchmark 使用了正确的类型（file_existence / content_correctness / functional_test / quality_standard / compatibility_verification）
