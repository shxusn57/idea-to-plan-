## 1. 执行顺序的生成方法

执行顺序是根据任务树中的依赖关系，通过拓扑排序算法生成的。同一批次内的 MEU 可以并行执行。

### 1.1 Kahn 算法拓扑排序（概要）

**为什么选择 Kahn 算法（BFS）而非 DFS 拓扑排序**：
- BFS（Kahn 算法）天然反映了并行执行的可能性——每一轮 BFS 处理的节点构成一个可并行执行的批次
- DFS 拓扑排序生成的是线性序列，无法直接体现并行关系

**算法关键步骤**（伪代码）：

```
1. 构建图：根据每个 MEU 的 dependencies 字段构建有向无环图（DAG）
2. 计算入度：每个 MEU 的入度等于其依赖的数量
3. BFS 逐层处理：
   - 每轮处理所有入度为 0 的 MEU（这些 MEU 的所有依赖都已满足）
   - 这些 MEU 构成一个可并行执行的批次
4. 更新入度：处理完一个 MEU 后，将其所有后继的入度减 1
5. 循环检测：如果处理完所有批次后仍有未处理的 MEU，说明存在循环依赖 → 报错
```

**注意**：上述算法只处理了显式声明的 `dependencies`。在实际使用中，还需要考虑父子依赖（隐式依赖）——子节点必须在父节点的所有子任务完成后才能开始。通常的做法是：先按分支层级排序（同一分支内的节点按拓扑排序），再在分支之间做拓扑排序。

### 1.2 排序规则

| 规则 | 说明 |
|---|---|
| **零依赖优先** | 入度为 0 的 MEU 放入第一批次 |
| **同批次可并行** | 同一批次内的 MEU 之间没有相互依赖，可以并行执行 |
| **循环依赖 = 错误** | 检测到循环依赖时立即报错，不生成执行计划 |
| **高风险优先** | 高风险的 MEU 应该尽早执行，以便留出足够的应对时间 |
| **关键路径优先** | 关键路径上的 MEU 优先安排在靠前的批次 |

## 2. 三层输出架构

执行计划的输出采用三层架构，每层服务于不同的消费者（人类或 AI），各层职责清晰、互不重叠。

### 2.1 三层职责分工

| 层 | 目录 | 主要消费者 | 核心职责 | 格式 |
|----|------|-----------|---------|------|
| **结构化数据层** | `framework/` | AI（程序化读取） | 任务定义、研究结论、验收标准等机器可解析的结构化数据 | YAML |
| **人类报告层** | `REPORT.md` | 人类（项目负责人、团队成员） | 执行计划总览、批次安排、风险评估、决策摘要等人类可读的报告 | Markdown |
| **执行指令层** | `prompts/` | AI（LLM 执行） | 每个 MEU 的精简执行 Prompt，包含上下文引用、避坑要点、验收引用 | Markdown |

**设计原则**：

- **framework/ 是唯一的数据源**：`prompts/` 中的 Prompt 不内联详细数据，而是通过引用指向 `framework/` 中的对应条目。这确保了数据一致性——修改 framework 中的数据后，Prompt 无需同步更新。
- **REPORT.md 是人类视图**：从 framework/ 中聚合生成，供人类阅读和评审。不作为 AI 执行的输入。
- **prompts/ 是执行入口**：每个 MEU 一个 Prompt 文件，AI 执行时加载对应 Prompt，再按引用按需读取 framework/ 中的详细数据。

> **按需深入**: [references/output-architecture.md](references/output-architecture.md)（三层输出架构详细规范）

## 3. 8 段式 Prompt 结构概览

每个 MEU 的执行 Prompt **必须包含且仅包含以下 8 个段落**。不允许省略任何一个段落，也不允许添加额外的段落。

> **设计理念**：精简 8 段式相比原 9 段式，移除了冗长的「分步执行指南」段落，改为通过上下文引用指向 `framework/` 中的结构化数据。Prompt 本身只包含执行 AI **必须立即知道**的关键信息（避坑要点、决策、约束），详细的技术上下文和执行步骤由 AI 按需从 framework/ 中加载。这大幅缩短了 Prompt 长度（约 60 行 vs 原 120 行），同时通过引用机制保证了信息的完整性和一致性。

| 段落 | 名称 | 核心作用 | 简要说明 |
|------|------|---------|---------|
| 第一段 | 背景 | 定位上下文 | 项目名称、批次号、已完成前置任务及关键产出（1-2 句话） |
| 第二段 | 任务 | 定义目标 | 目标描述（2-3 句话），不含范围和预期产出（这些在 framework/ 中） |
| 第三段 | 上下文引用 | 指向详细数据 | 引用 framework/ 中的 task-tree.yaml、research.yaml、benchmarks.yaml 对应条目 |
| 第四段 | 避坑要点 | 传递踩坑经验 | 从 research 的 pitfalls 提取最关键的 3-5 条（❌→✅ 格式） |
| 第五段 | 决策 | 记录技术决策 | 从 research 的 decision_summary 提取关键决策表（决策/选择/理由） |
| 第六段 | 约束 | 定义限制和规则 | 技术限制列表 |
| 第七段 | 产出 | 明确交付物 | 必须交付的文件列表 |
| 第八段 | 验收 | 引用验收标准 | 引用 benchmarks.yaml 中的具体条目（如 `- [ ] B1.1: 文件存在`） |

> **详细填写要求**：需要了解每段的详细说明、格式要求、示例和填写指南时，请读取 [step4-prompt-structure.md](references/step4-prompt-structure.md)。

## 4. 研究价值传导机制

Plan 的核心价值在于将 Research 阶段的发现**无损传递**给执行 AI。如果没有有效的传导机制，研究阶段的投入将白费——执行 AI 会在同样的坑上再次跌倒，做出已被否决的方案选择，忽略已知的边界条件。

### 4.1 信息传导映射表

以下是 Research 输出到 Plan 段落及 framework/ 结构化数据的完整映射关系：

| Research 输出 | Plan 中的目标段落 | 传导方式 |
|---|---|---|
| `decision_summary.key_decisions` | 决策（第五段） | 提取关键决策表（决策/选择/理由），详细决策信息存入 `framework/research.yaml` |
| `blockers[].pitfalls` | 避坑要点（第四段） | 提取最关键的 3-5 条（❌→✅ 格式），完整 pitfalls 存入 `framework/research.yaml` |
| `key_findings[category=caution]` | 避坑要点（第四段） | 映射为关键风险提醒，详细内容通过上下文引用指向 `framework/research.yaml` |
| `key_findings[category=approach]` | 上下文引用（第三段） | 通过引用指向 `framework/research.yaml` 对应条目，AI 按需加载 |
| `key_findings[category=api]` | 上下文引用（第三段） | 通过引用指向 `framework/research.yaml` 对应条目 |
| `key_findings[category=pattern]` | 上下文引用（第三段） | 通过引用指向 `framework/research.yaml` 对应条目 |
| `recommended_approach` | 上下文引用（第三段） | 映射为推荐方案描述，标注置信度，存入 `framework/research.yaml` |
| `confidence` | 上下文引用（第三段） | 在 research.yaml 中标注置信度（high/medium/low），低置信度结论在避坑要点额外提醒 |
| `open_questions[impact=blocking]` | 约束（第六段） | 映射为执行前必须解决的问题，如未解决则标记为阻塞 |
| `open_questions[impact=risk]` | 避坑要点（第四段） | 映射为风险提示，提醒执行者注意不确定性 |
| `references` | 上下文引用（第三段） | 存入 `framework/research.yaml`，供执行者需要时查阅 |

### 4.2 传导质量检查清单

在 Plan 生成完成后，应检查以下传导质量指标：

| 检查项 | 通过标准 | 不通过的后果 |
|---|---|---|
| Research 中的每个 `key_decision` 都有对应的决策条目 | 100% 覆盖 | 执行 AI 可能推翻已评估的方案 |
| Research 中的每个 `pitfall` 都已写入 `framework/research.yaml` 或 Prompt 避坑要点 | 100% 覆盖 | 执行 AI 会重复踩坑 |
| `confidence=low` 的结论是否在避坑要点有额外提醒 | 全部提醒 | 执行 AI 可能过度依赖不确定的结论 |
| `blocking` 级别的开放问题是否反映在约束条件中 | 全部反映 | 执行可能在阻塞问题未解决时开始 |
| `risk` 级别的开放问题是否反映在避坑要点中 | 全部反映 | 执行 AI 可能忽略已知风险 |
| Prompt 中的上下文引用是否正确指向 `framework/` 中的对应条目 | 全部可解析 | 执行 AI 无法加载所需的详细数据 |
| `framework/` 中的结构化数据与 Prompt 中的摘要信息一致 | 无矛盾 | 执行 AI 收到冲突信息，影响判断 |

### 4.3 传导失败的常见模式

以下是 Plan 生成中常见的传导失败模式，应当主动避免：

| 失败模式 | 表现 | 预防措施 |
|---|---|---|
| **信息丢失** | Research 发现了坑点，但 Prompt 和 framework/ 中均未提及 | 使用 4.2 的检查清单逐项核对 |
| **信息降级** | Research 中有详细的分析和来源，framework/ 中只写了结论 | 保留来源信息（如 Issue 编号、文档链接），便于执行者验证 |
| **信息错位** | 坑点信息放在了上下文引用中，而非避坑要点 | 严格按照 4.1 的映射表分配信息到对应段落 |
| **引用断裂** | Prompt 中引用了 `framework/research.yaml` 的条目，但该条目不存在或 ID 不匹配 | 生成后自动校验所有引用路径和条目 ID |
| **过度简化** | 决策条目只写了"选择了 X"，没有写理由 | 决策条目必须包含：决策、选择、理由 |
| **置信度缺失** | 研究结论没有标注置信度 | 所有来自 Research 的结论在 framework/ 中都必须标注置信度 |
| **数据不一致** | Prompt 中的摘要与 framework/ 中的详细数据矛盾 | 生成后比对 Prompt 摘要与 framework/ 数据的一致性 |
